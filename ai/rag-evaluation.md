# 📏 RAG 效果怎么评估？召回率、准确率、线上怎么验证

## 🤔 为什么 RAG 评估不能只看“准确率”？

很多团队在做 RAG（Retrieval-Augmented Generation）时，最先问的问题往往是：

```text
这个系统回答得准不准？
```

这个问题没错，但如果只盯着“最终答案准不准”，通常很难真正定位问题。

因为一个 RAG 系统至少包含两段核心链路：

```text
用户问题
   ↓
🔍 Retrieval：召回相关知识
   ↓
🧠 Generation：基于上下文生成答案
   ↓
📤 最终结果
```

所以 RAG 的问题通常也分成两类：

| 问题类型 | 典型表现 |
|------|------|
| 🔍 检索问题 | 没有召回到正确文档，模型“巧妇难为无米之炊” |
| 🧠 生成问题 | 明明文档已经找到了，但答案还是答偏、答漏、答错 |

这就是为什么评估 RAG 时，不能只看一个“大准确率”，而要拆开看：

- 🔍 检索层效果
- 🧠 生成层效果
- 🏁 端到端业务效果
- 🌐 线上真实表现

---

## 🧭 一套实用的 RAG 评估框架

如果从工程落地角度看，RAG 评估通常可以拆成三层：

```text
Layer 1：检索评估
看“有没有把对的内容找回来”

Layer 2：生成评估
看“拿到内容后有没有答对、答全、答得有依据”

Layer 3：系统评估
看“在线上到底有没有提升用户体验和业务指标”
```

可以把它理解成下面这个关系：

| 层 | 核心问题 | 代表指标 |
|------|------|------|
| 🔍 Retrieval | 找没找对 | Recall@K、Hit Rate、MRR、nDCG |
| 🧠 Generation | 答没答对 | Accuracy、F1、Faithfulness、Relevance |
| 🏁 System | 用户觉得好不好用 | 采纳率、追问率、人工转接率、延迟、成本 |

---

## 🔍 第一层：检索效果怎么评估？

RAG 的第一步不是回答，而是“找资料”。  
如果检索层做不好，后面的生成层再强也很难补救。

### 1. 📚 Recall@K（召回率）

这是 RAG 检索里最常见的指标之一。

它回答的问题是：

```text
在 Top-K 检索结果里，是否包含了正确文档？
```

例如：

- 某个问题的标准相关文档是 `Doc A`
- 检索系统返回 Top-5
- 如果 Top-5 里包含 `Doc A`
- 那么这个问题的 `Recall@5 = 1`

如果一批问题里有 80% 都能在 Top-5 里命中标准文档，那么：

```text
Recall@5 = 80%
```

这是一个非常关键的指标，因为它反映的是：

> 模型有没有机会看到正确知识。

### 2. 🎯 Hit Rate@K

Hit Rate@K 和 Recall@K 很像，在很多 RAG 场景里甚至可以近似理解。

它强调的是：

```text
只要 Top-K 里命中至少一个相关文档，就算成功
```

对于 FAQ、知识库问答、客服类场景，这个指标非常实用，因为很多问题只需要命中一条关键文档就够了。

### 3. 🥇 MRR（Mean Reciprocal Rank）

MRR 不只是关心“有没有找到”，还关心：

```text
正确答案排得靠不靠前
```

因为现实里大多数 RAG 系统不会把 Top-20 全都塞给模型，通常只会取前几条上下文。  
如果正确文档总是排在很后面，即使被召回了，也未必能真正起作用。

举个例子：

| 问题 | 第一条相关文档排名 | 倒数值 |
|------|------|------|
| Q1 | 第 1 名 | 1 |
| Q2 | 第 2 名 | 1/2 |
| Q3 | 第 4 名 | 1/4 |

把这些倒数值取平均，就是 MRR。

### 4. 📈 nDCG（归一化折损累计增益）

如果你的一个问题可能对应多条相关文档，而且不同文档相关程度不同，那么 nDCG 会比简单 Recall 更有用。

它更关心：

- 高相关文档是否排在前面
- 中低相关文档是否被合理排序

在复杂企业知识库、研究型问答、长文档检索场景里，这个指标更有参考价值。

### 5. 🧩 Precision@K（精确率）

Recall 关注“漏没漏”，Precision 关注“水分多不多”。

例如检索 Top-5：

- 其中 1 条是相关文档
- 4 条是噪音

那么虽然 `Recall@5` 可能是好的，但 `Precision@5` 会比较差。

这会带来一个实际问题：

```text
上下文里噪音太多，模型更容易被干扰
```

所以真正好的检索，不只是“找到了”，还要“找得干净”。

## 🧠 第二层：生成效果怎么评估？

检索只是把资料找回来，最终用户看到的是答案。  
所以生成层评估必须回答两个问题：

```text
答案对不对？
答案是不是基于检索内容得出的？
```

### 1. ✅ Accuracy / EM / F1

如果你的任务有标准答案，例如：

- FAQ
- 规则问答
- 封闭式知识题
- 数据集问答

那么可以直接用传统 QA 指标：

| 指标 | 含义 |
|------|------|
| ✅ Accuracy | 答案整体对不对 |
| 🎯 EM（Exact Match） | 是否和标准答案完全一致 |
| ⚖️ F1 | 在文本重叠层面兼顾精确率和召回率 |

但要注意，这类指标更适合“答案边界清晰”的任务。

### 2. 🧾 Faithfulness（忠实度 / Groundedness）

这是 RAG 场景里非常关键的指标。

它评估的是：

```text
模型的答案是否真正被检索到的上下文支持？
```

即使答案“看起来差不多”，如果它引用了上下文里没有的事实，那依然算问题。

例如：

- 检索内容只说“退款审核通常 3 到 5 个工作日”
- 模型却回答“退款会在 48 小时内到账”

这就属于典型的 **不 grounded**。

所以很多 RAG 评估体系里，Faithfulness 比表面准确率更重要。

### 3. 🎯 Relevance（相关性）

相关性看的是：

```text
答案是否真正回应了用户问题
```

有些答案本身并不虚构，但它可能绕开了用户真正关心的点。

例如：

- 用户问“会员退款规则是什么？”
- 系统回答了一大段“会员权益介绍”

这不一定是幻觉，但相关性明显不够。

### 4. 📚 Citation Quality（引用质量）

如果你的系统支持返回引用来源，那么还可以继续评估：

- 引用是否真的支撑当前答案
- 是否引用了最关键的文档
- 是否存在“引用看起来对，实际没支撑结论”的问题

在企业知识库、法律、医疗、金融等高风险场景里，这类指标非常重要。

## ⚖️ 第三层：端到端效果怎么评估？

很多团队离线指标做得不错，但上线后效果一般，原因就在于：

> 离线评估的是“模型质量”，线上考验的是“系统价值”。

所以除了检索和生成，还需要看端到端表现。

### 1. 🏁 Task Success Rate（任务完成率）

用户是否真的通过这次问答完成了任务？

例如：

- 是否成功找到政策说明
- 是否成功解决报错
- 是否成功完成下单/查询/排障

这比单独看“句子写得好不好”更接近业务目标。

### 2. 🙋 Follow-up Rate（追问率）

如果用户在得到答案后仍然频繁追问，通常说明：

- 首答不够完整
- 关键信息缺失
- 回答不够明确

所以追问率常常是线上非常有价值的反向信号。

### 3. 🧑‍💼 Escalation Rate（人工转接率）

在客服、IT 支持、企业助手场景里，可以重点看：

```text
用户最终是否还需要转人工？
```

如果一个 RAG 系统上线后人工转接率下降，往往比单纯的离线 Accuracy 更能证明系统价值。

### 4. 👍 Acceptance Rate（采纳率）

如果系统是面向内部员工、研发、运营或客服使用，还可以看：

- 用户是否复制了答案
- 是否点击了“采用”
- 是否把答案直接用于工单、回复、代码修改

这类指标本质上反映的是：

```text
用户是否真的相信并使用了这个答案
```

### 5. ⏱️ Latency / 💰 Cost

RAG 不是只看质量，还要看系统代价：

| 指标 | 说明 |
|------|------|
| ⏱️ P50 / P95 延迟 | 用户体感是否足够快 |
| 💰 单次问答成本 | 模型调用、检索、重排成本是否可控 |
| 📦 Token 消耗 | 上下文是否过长，是否影响性价比 |

很多时候，一个“略低一点但稳定便宜”的方案，线上反而更优。

## 🧪 离线评估应该怎么做？

如果你想系统化地评估 RAG，离线集是基础。

### 1. 📄 先构建评测集

一条好的 RAG 评测数据，最好包含：

| 字段 | 说明 |
|------|------|
| `question` | 用户问题 |
| `gold_docs` | 标准相关文档或片段 |
| `gold_answer` | 标准答案 |
| `category` | 问题类型，如 FAQ / 政策 / 操作步骤 |
| `difficulty` | 难度，如单跳、多跳、长文档、歧义问题 |

很多团队一开始就忽略这一步，直接拿几条示例测试，最后很难稳定比较不同方案。

### 2. 🔍 先单独评检索

不要一上来只看端到端答案。  
更稳妥的做法是先把检索模块单独拉出来评估：

- `Recall@1`
- `Recall@3`
- `Recall@5`
- `MRR`
- `nDCG`

这样你能先判断问题到底出在“没找到”还是“找到后没答好”。

### 3. 🧠 再评生成层

在固定检索结果的前提下，再去评估：

- 答案正确性
- 忠实度
- 相关性
- 引用质量

这样更容易定位是 Prompt 问题、模型问题，还是上下文组织问题。

### 4. 🪓 一定要做错误切片

不要只看总分。

同一个 RAG 系统，往往会在不同问题类型上表现差异很大：

- 单文档问答
- 多文档聚合
- 长政策文档
- 结构化数据问答
- 歧义问题
- 时间敏感问题

如果不做切片，很多关键问题会被平均值掩盖掉。

## 🌐 线上怎么验证 RAG 是否真的有效？

离线评估只能证明“理论上更好”，真正上线后，还要验证“用户是否真的受益”。

### 1. 🫥 Shadow Test（影子流量）

在正式切流前，可以先让新旧系统同时跑：

```text
线上真实请求
   ├── 旧系统返回给用户
   └── 新系统在后台跑结果做比对
```

这样不会影响用户，但可以先观察：

- 新系统召回率是否更高
- 新系统延迟是否变差
- 新系统是否更容易 hallucination

### 2. 🧪 A/B Test

如果系统已经比较稳定，最标准的做法还是 A/B：

- A 组走旧方案
- B 组走新方案
- 对比真实用户行为指标

重点看的通常不是一个指标，而是组合：

- 👍 点赞率 / 满意度
- 🙋 追问率
- 🧑‍💼 人工转接率
- ✅ 任务完成率
- ⏱️ 响应时间
- 💰 单次成本

### 3. 🛟 Canary Release（小流量放量）

对于高风险场景，不要直接全量切换。

更稳妥的路径通常是：

```text
1% → 5% → 10% → 25% → 50% → 100%
```

每一步都观察关键指标是否恶化，尤其是：

- 幻觉率是否上升
- 高价值问题是否答偏
- 延迟是否明显变差

### 4. 👀 人工抽检

线上指标很好，但仍然建议保留人工抽检。

因为很多严重问题并不会立刻体现在简单埋点里，例如：

- 答案语气过于自信
- 引用来源不匹配
- 关键数字答错
- 拒答策略异常

对于高风险业务，人工抽检是必须项，而不是可选项。

## 🚨 RAG 评估里最容易踩的坑

### 1. 只看最终答案，不看检索

这会导致你根本不知道问题出在 Retriever 还是 Generator。

### 2. 只看平均分，不做切片

平均分好看，不代表复杂场景表现好。

### 3. 只做离线评测，不做线上验证

离线集很容易和真实用户问题分布脱节。

### 4. 把“说得像”当成“答得对”

RAG 里最危险的问题不是“说得不好听”，而是：

```text
说得很像，但没有依据
```

所以 Faithfulness / Groundedness 必须单独评。

### 5. 忽略延迟和成本

一个质量提升 2%，但延迟翻倍、成本翻倍的方案，线上未必值得。

---

## ✅ 一套比较实用的落地建议

如果你的团队刚开始做 RAG 评估，可以先用下面这套最小方案：

### 离线先看这 4 个

- `Recall@5`：看标准文档有没有被召回
- `MRR`：看正确文档排位是否靠前
- `Answer Accuracy / F1`：看答案是否基本正确
- `Faithfulness`：看答案是否有上下文依据

### 线上先看这 5 个

- 👍 满意度 / 点赞率
- 🙋 追问率
- 🧑‍💼 人工转接率
- ⏱️ P95 延迟
- 💰 单次问答成本

### 发布流程建议

```text
离线评测
   ↓
影子流量
   ↓
小流量 A/B
   ↓
人工抽检
   ↓
逐步放量
```

这套方法不一定最复杂，但足够实用，也最容易在团队里落地。

---

## 🧭 一句话总结

RAG 评估不能只问“答案准不准”，而要拆成三个层面：

```text
检索有没有找对
生成有没有答对、答得有依据
上线后用户是不是真的更容易完成任务
```

所以一个成熟的 RAG 评估体系，通常至少要同时覆盖：

- 🔍 检索指标：Recall@K、MRR、nDCG
- 🧠 生成指标：Accuracy、F1、Faithfulness、Relevance
- 🌐 线上指标：满意度、追问率、转人工率、延迟、成本

只有把这三层拆开看，你才能真正知道：

> RAG 到底是“检索没做好”，还是“生成没做好”，还是“离线看起来不错，但线上并没有业务价值”。
